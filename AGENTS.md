# Repository Instructions

These instructions apply to this repository in addition to the global Codex instructions.

## Repository Layout

- `RegistryCI/` and `AutoMerge/` are separate Julia packages with separate `Project.toml` files and separate test suites.
- `docs/` is a separate Documenter project for the documentation and doctests.
- `example_github_workflow_files/` contains example workflow files, not the authoritative CI configuration for this repository.
- `AutoMerge/test/templates/` and `AutoMerge/test/cassettes/` contain test fixtures. Avoid incidental edits there unless the task specifically requires fixture updates.

## Validation

- Scope test runs to the package you changed whenever possible.
- For `RegistryCI` changes, prefer:
  `julia --project=RegistryCI -e 'using Pkg; Pkg.test()'`
- For `AutoMerge` changes that do not require integration coverage, prefer:
  `AUTOMERGE_RUN_INTEGRATION_TESTS=false julia --project=AutoMerge -e 'using Pkg; Pkg.test()'`
- For docs or doctest changes, use the `docs` project and run the relevant doctest or documentation build flow.
- If you cannot run a relevant check in the current environment, say so explicitly in the final handoff.

## Integration Tests

- `AutoMerge` integration tests are special-case and are not expected to run in routine local or sandboxed work.
- They require GitHub credentials and environment such as `AUTOMERGE_RUN_INTEGRATION_TESTS=true`, `AUTOMERGE_INTEGRATION_TEST_REPO`, and a GitHub token.
- CI intentionally skips real integration execution on ordinary pull request builds; they run from merge queue, push, or manual dispatch paths instead.

## Working Style

- Keep dependency, test, and debugging work scoped to the package you are modifying.
- Match existing CI and documentation workflows rather than inventing new local conventions.
- Add yourself as a co-author to every commit that you make.
- PR titles for single-package changes should begin with `RegistryCI: ` when the PR modifies only `RegistryCI/`, or `AutoMerge: ` when the PR modifies only `AutoMerge/`.
- Do not require either prefix when a single PR modifies both packages, though single-package PRs are preferred when practical.
- Do not require either prefix for repo-maintenance or meta-only PRs that modify neither package, such as changes to `AGENTS.md`.
- Pull requests must include a sentence that discloses the AI authorship, for example: `🤖 Generated by Codex.`
